# Codex Round v0.7.0-r1 — Opportunistic Current Site Discovery

Date: 2026-06-26
Subject: review of v0.7.0 design (Current Site discovery sub-protocol replacing "ask 4 values upfront")
Source: real user screenshot — user asked "帮我打开后台", AI refused because PROJECT_INDEX.md Current Site had empty fields, asked for 4 values including a constant (workspace_url) and auto-detectable fields (browser_profile). User: "不应该是直接打开 workspace.laicms.com? 没登录引导登录,然后看几个站点 — 1 个直接进,多个问下?"

## Findings + responses

### Fprocess.1 [high] discovery vs first-contact Phase 1/2 state-machine collision
- Status: **applied**
- Resolution: `current-site-discovery.md` opens with explicit **Preconditions** section. If Phase 2 is in progress, discovery's Step 4-5 are inlined into Phase 2 (no independent run). If Phase 2 ended and PROJECT_INDEX has at least `site_id` filled, discovery resumes from the first unfilled field (no re-opening of login page). SKILL.md Hard Gate wording matches.

### Fclassification.1 [high] silent-write to PROJECT_INDEX violates v0.6 "必须报告" whitelist
- Status: **applied**
- Resolution: Step 1 of `current-site-discovery.md` requires AI to inline-report the auto-fill in the same turn ("workspace_url 默认填 https://workspace.laicms.com/ (AllinCMS SaaS 标配); self-hosted 请打断我"). `first-contact.md` § Don't narrate updated: discovery-stage Current Site constant writes ARE on the report whitelist (writing files always was).

### Fclassification.2 [med] MINOR bump boundary for pin users
- Status: **applied**
- Resolution: kept as MINOR. CHANGELOG v0.7.0 explicitly notes "pin@0.6 用户不受影响 — discovery 仅在 unpinned 或 ≥0.7 时生效".

### Fprocess.2 [med] Step 3 wait/resume contract under user multi-message reply
- Status: **applied**
- Resolution: Step 3 adds **Resume contract**: any subsequent user message containing a login-confirmation signal (`好了 / done / login done / 登好了 / 已登录`) triggers Step 4 first, even if other content follows in the same message. Original request stays pending until discovery completes.

### Ffalsifiability.1 [med] DOM-scraping judgments will break on workspace redesign
- Status: **applied**
- Resolution: each discovery step requires **≥ 2 independent signals** (URL pattern + DOM selector + page title — any two). `audits/discovery-fixtures/dashboard-snapshot.md` lays out the expected HTML structure for the current workspace.laicms.com — used as a regression baseline. `discovery_failure_count` counter: after 2 consecutive Step 3 failures, fall back to the v0.6 "ask 4 values" path with a one-line explanation.

### Fhistorical.1 [med] self-hosted users have no opt-out today
- Status: **applied**
- Resolution: new optional PROJECT_INDEX field `deployment: saas | self-hosted` with default `saas`. Discovery Step 1 reads this field; only opens workspace.laicms.com if `saas`. For self-hosted, AI asks the workspace URL once and writes it.

### Fprocess.3 [low] tooling-matrix phrasing conflates "open backend" and "publish"
- Status: **applied**
- Resolution: split into two `user_facing_phrasing` rows. (a) `current_site × open_backend_or_monitor`: "打开后台 / 跑监控时我会先开 workspace 引导登录确认，自动补 ID — 不用手填". (b) `current_site × publish`: "Current Site 字段会在第一次开后台时由 discovery 自动补上;不用现在手填". Same intent, accurate split.

## Verification

- discovery flow respects v0.6's preamble + don't-narrate rules (reports auto-fills, doesn't narrate Phase or sub-protocol entry).
- discovery has explicit fallback to v0.6 "ask 4 values" after 2 failures — backward-safe.
- pin@0.6 users see no behavior change.
- self-hosted users get an opt-out via the `deployment` field, defaults to SaaS (no change for the majority).
